# 快速修复 Token 错误

## 🎯 问题现象

- 第一次请求成功 ✅
- 第二次请求失败 ❌ "获取临时token失败"

## 💡 可能的原因

### 1. API 限流
智谱 AI 可能对请求频率有限制，连续快速请求可能被限流。

### 2. Token 缓存问题
Token 缓存机制可能导致第二次请求时出现问题。

### 3. 并发请求问题
多个请求同时调用 getToken 可能导致冲突。

## ✅ 已应用的修复

### 修复：改用箭头函数

**修复前：**
```typescript
async getToken(): Promise<string> {
    // ...
}
```

**修复后：**
```typescript
getToken: async (): Promise<string> => {
    // ...
}
```

**原因：** 箭头函数可以避免 `this` 绑定问题

## 🔧 立即尝试的解决方案

### 方案 1：清理缓存并重试

```
1. HBuilderX -> 运行 -> 清理项目缓存
2. 重新运行项目
3. 等待 2-3 秒后再发送第二条消息
```

### 方案 2：切换到更快的模型

编辑 `uni_modules/uni-ai-x/config.uts`：

```typescript
const defaultLLM: DefaultLLM = {
    provider: 'zhipu',
    model: 'glm-4-flash'  // 改为 flash，更快更稳定
}
```

**优点：**
- 响应更快
- 更稳定
- 成本更低

### 方案 3：添加请求延迟

如果是限流问题，可以在发送消息时稍微等待一下：

```
发送第一条消息 -> 等待 2-3 秒 -> 发送第二条消息
```

## 🔍 调试步骤

### 步骤 1：查看控制台日志

打开浏览器控制台（F12），查找：

```
智谱 AI - 获取 token: 33a5b411... Time: 1234567890
```

**检查：**
- 第一次请求是否有这条日志？
- 第二次请求是否有这条日志？
- 两次请求的时间间隔是多少？

### 步骤 2：查看 Network 请求

在 Network 标签中：

1. 查看第一次请求的状态码（应该是 200）
2. 查看第二次请求是否发送了
3. 如果发送了，查看状态码和响应

**可能的状态码：**
- `200` - 成功
- `429` - 请求过于频繁（限流）
- `401` - 认证失败
- `500` - 服务器错误

### 步骤 3：检查错误详情

在控制台中，点击错误信息，查看完整的错误堆栈。

## 📊 常见错误码及解决方案

| 错误码 | 含义 | 解决方案 |
|--------|------|----------|
| 429 | 请求过于频繁 | 等待几秒后重试 |
| 401 | API Key 无效 | 检查 API Key |
| 500 | 服务器错误 | 稍后重试 |
| 503 | 服务不可用 | 检查智谱 AI 服务状态 |

1. 发送第一条消息："你好"
2. 等待响应完成
3. **等待 3 秒**
4. 发送第二条消息："你好呀"
5. **结果：** 应该成功 ✅

1. 发送第一条消息："你好"
2. **立即**发送第二条消息："你好呀"
3. **结果：** 可能失败 ❌（限流）

## 💡 最佳实践

### 1. 避免快速连续请求

```
✅ 好的做法：
发送消息 -> 等待响应 -> 等待 1-2 秒 -> 发送下一条

❌ 不好的做法：
发送消息 -> 立即发送下一条
```

### 2. 使用合适的模型

```
日常对话 -> glm-4-flash
简单问答 -> glm-4-air
复杂任务 -> glm-4-plus
```

### 3. 监控 API 使用

- 定期检查智谱 AI 控制台
- 查看 API 调用次数
- 监控余额使用
